Relatório de Estratégias de Produção para o conciliacao_app

Data da análise: 13/03/2026

Escopo: análise técnica comparativa sem alterações no projeto.

Resumo Executivo

O app atual (Rails 8.1, sem banco, processamento síncrono e uso de tmp/conciliacao/<uuid>) é compatível com entrada rápida em plataforma gerenciada desde que exista disco persistente local por instância. O principal risco atual é escalar horizontalmente sem storage compartilhado/sessão consistente, além de timeout em requests longas.

Para perfil de baixo custo inicial + operação gerenciada, a recomendação é iniciar com Render (alternativa direta: Railway) em arquitetura de 1 instância com disco, endurecer operação e evoluir para storage externo + processamento assíncrono antes de buscar alta escalabilidade/HA.

1) Diagnóstico do projeto para produção

Requisitos operacionais derivados do código

  • Upload e artefatos salvos em filesystem local temporário: tmp/conciliacao/<uuid>.
  • Fluxo de múltiplas etapas depende de sessão com paths locais (/upload -> /configurar -> /processar -> /download).
  • Sem banco/ActiveRecord no app; estado operacional está em sessão + disco local.
  • Processamento ocorre de forma síncrona dentro da request HTTP.
  • Não há rotina explícita de limpeza de tmp/conciliacao.
  • Em múltiplas réplicas, há risco de perda de continuidade quando requisições caem em instâncias diferentes.
  • TLS no Rails ainda não está forçado por padrão em produção (force_ssl/assume_ssl comentados).
  • Observabilidade inicial existe via logs STDOUT e healthcheck /up, mas sem alertas/métricas operacionais mínimos.

Checklist obrigatório antes do go-live

  • Definir limites de upload (tamanho, tipo de arquivo, validação de extensão/mime).
  • Implementar limpeza automática de temporários por TTL.
  • Definir retenção de arquivos e política LGPD (dados financeiros sensíveis).
  • Configurar timeout de aplicação abaixo do timeout da plataforma/proxy.
  • Habilitar HTTPS forçado em produção quando houver proxy SSL.
  • Configurar logs estruturados e rastreio por request_id.
  • Definir métricas/alertas: CPU, memória, disco, erro 5xx e latência p95/p99.
  • Executar teste de concorrência com uploads/processamentos simultâneos.
  • Validar rollback real em staging antes de produção.

2) Comparativo de estratégias (6 opções)

2.1 Render (Web Service + Disk)

  • Arquitetura alvo: 1 web service + disk persistente anexado.
  • Esforço de setup: baixo.
  • Custo inicial estimado: US$ 9–15/mês (Starter + disk básico).
  • Escalabilidade: boa para stateless; limitada para este app com disk local.
  • Confiabilidade: boa no single-node; restrições para zero-downtime com disk.
  • Segurança/compliance: TLS gerenciado, base gerenciada sólida.
  • Lock-in: médio.
  • Riscos para este app: filesystem local e sessão/afinidade podem quebrar em múltiplas instâncias.
  • Quando usar: MVP de baixo custo e alta simplicidade.
  • Quando evitar: necessidade imediata de HA ativa-ativa.

2.2 Railway (container managed)

  • Arquitetura alvo: serviço web com volume persistente.
  • Esforço de setup: baixo.
  • Custo inicial estimado: US$ 5–15/mês (Hobby + uso + volume).
  • Escalabilidade: com volume há limitações para múltiplas réplicas.
  • Confiabilidade: boa para 1 réplica; pode ter breves interrupções em redeploy com volume.
  • Segurança/compliance: SSL automático e operação gerenciada.
  • Lock-in: médio.
  • Riscos para este app: mesma dependência de storage local e continuidade de sessão.
  • Quando usar: MVP econômico com setup rápido.
  • Quando evitar: necessidade de escalar horizontalmente sem refatorar estado.

2.3 Fly.io (Machines + Volume)

  • Arquitetura alvo: machine(s) com volume local por máquina.
  • Esforço de setup: médio.
  • Custo inicial estimado: US$ 6–14/mês (compute pequeno + volume).
  • Escalabilidade: flexível, inclusive multi-região; volume requer estratégia de dados.
  • Confiabilidade: média-alta quando bem operado.
  • Segurança/compliance: boa, com maior responsabilidade de operação.
  • Lock-in: médio.
  • Riscos para este app: paths locais/sessão local não compartilhados entre machines.
  • Quando usar: custo baixo com maior controle técnico.
  • Quando evitar: equipe sem tempo para operação mais fina.

2.4 Heroku (dyno/container)

  • Arquitetura alvo: dyno web stateless.
  • Esforço de setup: baixo.
  • Custo inicial estimado: a partir de US$ 7/mês (sem incluir storage externo).
  • Escalabilidade: simples para apps stateless.
  • Confiabilidade: boa para desenho stateless.
  • Segurança/compliance: plataforma madura.
  • Lock-in: médio-alto.
  • Riscos para este app: filesystem efêmero + timeout de request tornam o fit ruim sem refatoração.
  • Quando usar: após mover arquivos para storage externo e reduzir síncrono longo.
  • Quando evitar: cenário atual sem mudanças.

2.5 Kamal + VPS (Hetzner/DO/Linode)

  • Arquitetura alvo: container em VPS via Kamal, com disco local/volume.
  • Esforço de setup: médio-alto.
  • Custo inicial estimado: US$ 4–12/mês (single VPS), maior em HA.
  • Escalabilidade: alta flexibilidade, depende de operação do time.
  • Confiabilidade: potencialmente alta com práticas maduras.
  • Segurança/compliance: maior responsabilidade do time (hardening, backup, patching).
  • Lock-in: baixo.
  • Riscos para este app: risco operacional (monitoramento, backup, failover) mais do que técnico.
  • Quando usar: busca por controle e baixo lock-in.
  • Quando evitar: prioridade total em operação gerenciada.

2.6 Cloud Run / App Runner (serverless container)

  • Arquitetura alvo: container stateless com escala automática.
  • Esforço de setup: médio no deploy; alto para adequar o app atual.
  • Custo inicial estimado: Cloud Run US$ 0–20/mês (baixo uso), App Runner US$ 5–30+/mês.
  • Escalabilidade: excelente.
  • Confiabilidade: alta quando app é stateless.
  • Segurança/compliance: forte para ambientes enterprise.
  • Lock-in: médio-alto.
  • Riscos para este app: storage local efêmero e fluxo síncrono exigem refatoração.
  • Quando usar: fase de alta confiabilidade/escala após desacoplamento de disco local.
  • Quando evitar: lift-and-shift sem mudanças.

4) Plano de evolução sem retrabalho

Fase 1: Plataforma gerenciada de entrada

  • Iniciar em Render (ou Railway) com 1 instância + disk/volume.
  • Aplicar checklist mínimo: limite de upload, TTL de arquivos, TLS e observabilidade básica.

Fase 2: Endurecimento operacional

  • Centralizar logs, criar alertas e painéis operacionais.
  • Definir backup/restore de temporários conforme retenção.
  • Formalizar limites de concorrência, timeout e políticas de retenção.

Fase 3: Migração para maior robustez

  • Externalizar arquivos para object storage (S3/R2/GCS ou equivalente).
  • Substituir sessão com path local por identificador de job/execução.
  • Mover processamento pesado para background e reduzir caminho síncrono.
  • Escolher trilha final: Kamal+VPS (controle/custo) ou Cloud enterprise (gerenciado/escala).

Validação aplicada ao relatório

  • Cada estratégia cobre explicitamente: deploy, domínio/TLS, persistência de temporários, escala e rollback.
  • Custos e limites alinhados com documentação/preços oficiais atuais (13/03/2026).
  • Trade-offs contextualizados ao comportamento real do app: upload + processamento síncrono + download.
